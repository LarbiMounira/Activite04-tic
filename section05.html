Chapitre 5: système de gestion de version

Introduction

Dans ce chapitre, nous allons voir pourquoi Git est si populaire et pourquoi les systèmes de gestion de version sont incontournables aujourd'hui.

Qu’est-ce que Git ?

Git est un logiciel créé en 2005 par LINUS TORVALDS qui garde l'historique des modifications et changements sur un projet, maintenant il est obligé à chaque développeur de savoir l'utiliser qui prouve son importance.

A quoi sert concrètement un système de gestion de version ?

Les grands projets sont réalisés par plusieurs développeurs, donc il y a toujours un risque de conflits et de mauvaise organisation entre ces développeurs en raison des changements nécessaires ou de la nécessité de tester de nouvelles fonctionnalités qui posent des problèmes, donc la solution était de mettre en place un serveur distant pour stocker l'historique des modifications et permettre à chaque développeur de travailler sur une copie locale.

Modèle centralisé VS modèle décentralisé :

Modèle centralisé : signifie que le code source d'un projet est stocké sur un serveur distant centralisé et pour y accéder, il faut se connecter à ce serveur.
Modèle décentralisé : le code source est toujours stocké sur un serveur distant, mais chaque utilisateur doit télécharger et héberger le code source sur sa propre machine. Ce modèle a été popularisé grâce à :
1) Simplicité / flexibilité du travail.
2) La sécurité.

Qu’est-ce que GitHub ?

GitHub est un service en ligne ou une plateforme de développement collaboratif de logiciels qui facilite la gestion de projet grâce à ses fonctionnalités, telles que la tenue d'une trace des modifications apportées à un projet, et il facilite également le travail en groupe car il est public, ce qui signifie que les utilisateurs peuvent stocker et gérer du code source, ainsi que collaborer avec d'autres développeurs sans aucun problème.

La gestion des informations selon Git

Git stocke les données sous forme d'instantanés ou de "snapshots". Chaque instantané est enregistré localement dans une base de données sur notre propre machine, ce qui permet de réaliser la plupart des opérations localement sans avoir besoin de se connecter à un serveur central distant. Cela rend les opérations plus rapides et facilite le travail.

Les états des fichiers

Un fichier peut avoir deux états dans Git : sous suivi de version ou non suivi.
Il peut être Suivi : s'il est enregistré en base et non suivi sinon.
Lorsque nous importons un répertoire existant sur notre machine pour le démarrage d'un dépôt Git, les fichiers ne sont pas suivis et nous devons les indexer et les valider. En revanche, lorsqu'on clone un dépôt Git existant, tous les fichiers sont suivis par défaut donc enregistrés et validés (stockés dans la base de données locale). Chaque fichier suivi peut avoir l'un des trois états suivants : modifié, indexé ou validé.
Dans nos projets, l'utilisation des fichiers est certaine, donc chaque fichier utilisé est considéré comme un fichier "modifié ou non suivi", et lorsque nous terminons, nous indiquons à Git d'enregistrer les fichiers et ils seront parmi les fichiers indexés ou non modifiés.

Les zones de travail:

Tout projet Git est composé de trois sections : le répertoire de travail, la zone d'index et le répertoire Git.
Le répertoire de travail correspond à une extraction unique de la base de données dans le répertoire Git et est placé sur le disque pour être utilisé ou modifié.
La zone d'index correspond à ce qui est situé dans le répertoire Git, qui stocke les informations concernant ce qui fera partie du prochain "commit".
Le répertoire Git est l'endroit où Git stocke la base de données des objets de votre projet. C'est le cœur de Git.
Ainsi, le processus de travail sera : nous allons travailler sur nos fichiers dans le répertoire de travail. Lorsqu'on modifie ou crée un fichier, on peut ensuite choisir de l'indexer. Lorsque le fichier est nouveau ou est modifié, il n'est pas indexé tant que son nomn'est pas ajouté à la zone d'index.
Finalement, on va valider ("commit") la version indexée de nos fichiers pour les ajouter au répertoire Git.

Lorsqu'on utilise "git init", pour confirmer que le dépôt Git a bien été initialisé et qu'il est vide.
On peut utiliser ici la commande "git status" pour déterminer l'état des fichiers de notre répertoire.
Pour indexer des fichiers, on utilise la commande "git add." On peut lui passer un nom de fichier pour indexer le fichier en question, le nom d'un répertoire pour indexer tous les fichiers du répertoire.
Remarque : les caractères comme * et ? vont posséder une signification spéciale. Lorsque j'écris "git add *.txt", je demande à Git d'ajouter à l'index tous les fichiers du projet qui possèdent une extension .txt.

Lorsqu'on utilise "git commit" sans argument, une nouvelle fenêtre s'ouvre en utilisant dans la majorité des cas VIM.
Si votre éditeur est bien VIM, il faut appuyer sur la touche "esc" pour sortir du mode insertion puis taper :wq et entrée pour valider et quitter.

14) Récupérer et envoyer des fichiers depuis GitHub

On utilise ces commandes pour cloner un repo sur GitHub vers notre PC et récupérer des fichiers d'un projet déjà existant. Il suffit de copier le lien du repo, utiliser la commande "git clone" et coller le lien.
Pour récupérer des modifications apportées au repo distant (GitHub) par d'autres utilisateurs, on utilise la commande "git pull".
Enfin, pour envoyer nos modifications locales (sur notre PC) au repo distant, on utilise la commande "git push".
NB : Git est un logiciel de gestion de versions distribué, donc chaque utilisateur a une copie complète de l'historique des modifications sur son PC, ce qui signifie que chaque utilisateur peut récupérer les modifications apportées par les autres utilisateurs à condition que les commandes "pull" et "push" soient correctement exécutées.

Conclusion

En conclusion, Git est un système de gestion de version très populaire qui permet de garder une trace des modifications apportées à un projet et facilite le travail collaboratif. GitHub, quant à lui, est une plateforme de développement collaboratif qui facilite la gestion des projets Git. Apprendre à utiliser Git et GitHub est essentiel pour les développeurs d'aujourd'hui afin de gérer efficacement les projets et collaborer avec d'autres développeurs.
